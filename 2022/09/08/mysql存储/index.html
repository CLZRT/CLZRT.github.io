Mysql基本结构

引入

我们在使用mysql时,总是需要创建表,再向表插入数据.以便更好地组织数据. 语句如下

在使用时我们总是会好奇说, 创建表时,我们表结构数据存放到哪去了? 插入数据时,也不免对mysql如何存储数据感到好奇.

用户记录存储

我们把向MySQL中插入的一条条数据称为记录.

不同的存储引擎对记录的存储格式不同 (存储引擎负责对数据的读写)

在Innodb引擎中对记录的存储格式分为

Compact(主讲)

Redundant(稍微讲讲)

Dynamic(略

Compressed (略

各个行格式需要解决的问题是,如何区分各个列?

Compact

首先是Compact格式,它将单条记录分为了4个部分 分别是

变长列长度 :

​ 单条记录中,可变列的长度 (用于区分真实数据中各个列的位置) 逆序排放

空列位置 :

​ 记录记录中,数据为空(null)的列, 节省空间 逆序排放

记录头信息 :

记录下一条记录的位置,标记记录是否被删除,

B+树的每层非叶子节点的最前面一个的标记,

一个页面中 的某个组内的最后一条数据,

当前记录在页面堆中的相对位置,记录类型

下一条记录的相对位置

真实数据 :

存放我们插入的数据

Redundant

再是 Redundant 格式

每一列的偏移长度

记录头信息

真实数据

讲了半天记录的存储格式,你也妹讲,它到底存在哪里啊?

这些记录都存储于索引页中,我们查找数据也是去索引页中查找,而页也是mysql从磁盘中读取数据的最小单位.页的大小固定为16kb.

页

我们知道页是存放用户记录的基本数据结构(其实单指索引页),那么页之间是如何做好区分?页与也之间是如何连接起来的?我们查找数据是如何知道这个页里有我们想要的数据呢? 在页中我们如何快速地找到我们的数据记录呢?

围绕这几个问题,我们将了解到(索引)页的结构.

页的结构

File Header

MySQL中,页与页之间使用页号进行区分,而页号就存与页的File Header部分,此外,File Header还存放 上(下)两个个页的页号,实现页与页之间的连通.

总之页之间通用的信息都放在这里(页号,页类型,页属于空间,页校验和 checksum)

Page Header

Page Header 为索引页独属部分,用于标记索引页所特有的信息,如

页目录的槽数量

已删除记录的头节点在页面中的偏移量 :方便重新利用已删除记录的空间

用户记录数量

索引id:表示属于哪一个索引

B+树非叶子节点段(top)的根节点标记: 仅在根节点页面才定义**(存放Segment Header结构)**

Segment Header

  1. INODE Entry 结构所在表空间id
  2. INODE Entry 结构所在页页号
  3. INODE Entry 结构在该页中的偏移量

B+树叶子节点段(leaf)的根节点标记: 仅在根节点页面才定义**(存放Segment Header结构)**

Infimum+Supremum

Infimum: 系统定义的第一条记录,不存储用户数据

Supremum: 系统定义的最后一条记录,不存储用户数据

tips: 记录顺序不以物理位置排序,以 主键(聚簇索引)大小排序.

User Records

用户记录存储位置

Free Space

尚未分配给用户记录的空余位置

Page Directory

在页中,将多条数据组织为一组,用于减少在页中遍历记录的数量.(假设,页中一百条数据,我们要取 第67条.根据页的结构我们可以从Innimum往后取第67条数据)

而我们完全可以将100条记录分为10组,每组10条,我们就可以直接定位到第六组的第七条,在这过程中,我们只需要遍历7次(组号:1次,组内记录:6次)

为此,我们需要在页中的 搞一个 各个组的目录,用于定位到各个对应的组, (我们将每个组的最后一条记录在页中的偏移量,放于 页目录的槽中)

File Trailer

用于校对页的完整性(配合File Header)

总结

相信到这里,上面的问题,已经全部了解了.

页之间是如何做好区分?

答:页号

页与也之间是如何连接起来的?

答:文件头,的上下页号

我们查找数据是如何知道这个页里有我们想要的数据呢?

答:还不知道,我们无法在多个页中找到记录所在的页. 这是我们下一节要涉及的内容,卖个关子.

在页中我们如何快速地找到我们的数据记录呢?

答: 利用Page Directory 与 记录中的记录头信息(组内大哥:存放组内记录数量,组内小弟:0)

索引

ok,终于来到索引了… MySQL中的索引指的是 以某一列 为排序列而形成对用户记录的有序组合(链表[组] B+树)

有了这个有序组合(树), 我们就可以快速定位到我们查询数据所在的(索引)页, 之后就是根据Page Directory 找到对应的组巴拉巴拉…

为此,我们将索引分为两个部分. 一个是叶子节点,用于存放 我们最后所要得到的数据, 一个是非叶子节点,用来组织下层节点.

那我们B+树的根节点,在MySQL的哪里呢?

记得前面页面结构里的Page Header吗?

里面对索引的非叶子节点和叶子节点的根节点做了标记… 所以我们的根节点存放于一个普通页中(位置普通,在表空间中),只不过其被做了标记..

想想也是,一个表中索引的数量并不是确定的,用户想创建多少就创建多少,而且索引的指代,已经在表空间的第一个区的第三个页面完成了, 索引的根节点,也就没必要特殊对待了

区,组与段

区

现在,我们有了页,有了索引. 逻辑上说,我们可以快速地检索到我们数据了. 但我们知道,机械硬盘中,是以区为单位划分数据的,不同区内的数据,需要机械硬盘移动磁头获取,而这需要很多时间. 为此我们需要尽量地把相邻地数据(索引页)放在同一个磁盘区内, MySQL也就因此诞生了MySQL的区的概念

MySQL的区 是由64个页组成,一个索引页所占空间为16kb(2^4kb),所以 一个区占2^10kb=2mb

组

而组的出现,也仅仅是将表空间的各个区组织起来,便于MySQL管理 (因为表空间中,区的数量实在太多了)

段

段的出现也是类似于组,但段是为了指代索引的真实物理存储而存在的. 怎么理解呢?

表空间指代的是MySQL中的某一个数据表, 而MySQL要读取数据表的索引信息就需要读取INODE Entry(位于表空间第一个区的第三个页面,若段数目>85个,会占用后面的页面) ,在第一个页面可以获取 INODE页链表头节点**[第一个组]**

而Innodb Entry 所指向的/区就是索引(段) [索引由其叶子节点段与非叶子节点段组成]

INODE Entry 与 XDES Entry

INODE Entry(存放于第一个组)

XDES Entry(存放于每一个组的第一个页面)

Segment ID : 如果,区被分配给某一个段,则其存放所属段的id

表空间中存在的其他页面

FSP_HDR类型页(只存在于第一个组的第一个页面)

表空间的第一个页面,存放表空间的一些整体属性,和所在的第一个组的256个区的 XDES Entry结构,比较重要地是 其中的File Header部分**[存放表空间整体属性的部分]**

XDES类型页(存在于每个组的第一个页面)

类似于FSP_HDR页 (FIle Space Header 换为 Not Used)

IBUF_BITMAP 类型(存在于每个组的第二个页面)

对页进行修改,需要同步对各个索引进行修改,而这会产生很多随机IO(各个索引分布位置随机), 所以产生 Change Buffer 缓存变化,等到该索引被其他原因(or服务器空闲)再从change(insert) buffer写入
